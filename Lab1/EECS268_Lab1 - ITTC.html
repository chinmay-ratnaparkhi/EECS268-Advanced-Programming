Overview

Let's make a console based video game! This is a two player game where two pokemon are created and fight each other to the death or to the draw. Each player will get to define the name, hp, attack level, and defense level for their pokemon. After each player makes a monster, randomly decide who goes first using a 2-sided die, then they will fight for a maximum of 10 rounds. If either player dies during the game, the game must stop. If they are both standing at the end, it's a draw.

Glossary We'll not only be using dice in this lab, we'll be using nerdy terminology to refer to them. d20: a 20 sided die (1-20) d6: a 6 sided die (standard die with 1-6) d8: an 8 sided die (1-8) d2: a 2 sided die (1 or 2)

Makefile

  • I highly recommend that you make your make file compile just your main.cpp, test it, then get it to compile the main and the Dice code, test it, add a couple more files and so on...

  • If you've added a loop, a control statement, or a simple function, stop! Test it!

Pokemon Class

Private Member Variables

  • int m_hp, the health points of the pokemon
  • int m_attackLevel, the attack level of the pokemon
  • int m_defenseLevel, the defense level of the pokemon
  • string m_name, name of the pokemon


Public Methods

  • Pokemon()
    • Constructor
    • Initializes all numeric data to zero and strings to ""


  • void reduceHP(int amount)
    • Reduces m_hp by the amount
    • Does not prevent m_hp from becoming a negative value
  • Create get and set methods for HP, attackLevel, defenseLevel and name
  • All get methods should be const.

Example 

//in header file
int getHP() const;

//in cpp file
int Pokemon::getHP() const
{
   return(m_hp);
}

Colosseum class

private members:

  • Dice d20
  • Dice d6

public members:

  • Colosseum()
    • Initializes Dice to appropriate sizes
      • e.g. d20 = Dice(20);


  • void userBuild(Pokemon& p)
    • sets all the values for name, hp, attackLevel, and defenseLevel or the Pokemon passed in
    • the hp must be between 1-50
    • the attackLevel+defenseLevel must be between 1-50
    • you must keep prompting the user for good input


  • bool attack(const Pokemon& attacker, Pokemon& defender)
    • The attacker is trying to hit the defender
    • Calculate attack and defense bonuses:
      • use d20 to roll random values for and attackBonus and a defenseBonus
    • Determine if the attack hits
      • if the attacker's attackLevel + attackBonus is greater than the defender's defenseLevel + defenseBonus the attack hits
    • Determine how much damage the attack does (if it hit)
      • roll three d6's, add them up, and that is the damage
    • Displays information about the attack
      • display all attack information as reflected in the sample runs
    • Determine if the player died from the attack
      • return true only if the opponent died because of the attack, otherwise returns false
  • void play(Pokemon& p1, Pokemon& p2)
    • Handles the 10 round fight between p1 and p2
    • Uses a 2-sided die to decide who goes first
    • Display round information and calls attack() to display attack information
    • Determines when the game is over

main.cpp

  • int main()
    • declares two Pokemon, one Colosseum and any other variables you see fit
    • Displays announcement to players to build their pokemon then...
    • uses the Colosseum method to handle the building of pokemon
    • calls play from the Colosseum class, passing in the two pokemon
    • Repeat the building and playing of the game until the user wants to quit (see sample run)
      • Note: you'll be reusing the same instances of pokemon, but the userbuild method will redefine their values

How to use an input.txt file

In the provided input.txt file, you can pass all the input to your program as if you were typing it in by hand. Why?

  • It's faster than manual input
  • You can make custom test file for certain scenarios

To use the file, from the console run: 

yourLabDirector> ./Lab01 < input.txt

Note: since the input is coming from a file, you won't see it displayed in the console but it is being fed into the program.

Submission instructions

Unless it your project is graded during the lab, send a tar file with all the necessary files (*.h, *.cpp, and makefile) in a tar file to your GTA email. Do not include any object (*.o) or executable file.

Creating a File Archive Using Tar

The standard Unix utility for created archived files is tar. Tar files, often called tarballs, are like zip files.

  1. Create a folder to hold the files you will be sending in. The folder should be named like LastName-KUID-Assignment-Number:
    mkdir Smith-123456-Lab-01
  2. Now, copy the files you want to submit into the folder:
  3. Tar everything in that directory into a single file:
    tar -cvzf Smith-123456-Lab-01.tar.gz Smith-123456-Lab-01
    That single command line is doing a number of things:
    • tar is the program you're using.
    • -cvzf are the options you're giving to tar to tell it what to do.
      • c: create a new tar file
      • v: operate in verbose mode (show the name of all the files)
      • z: zip up the files to make them smaller
      • f: create a file
    • Smith-123456-Lab-01.tar.gz: the name of the file to create. It is customary to add the .tar.gz extension to tarballs created with the z option.
    • Smith-123456-Lab-01: the directory to add to the tarball

Please note that it is your responsibility to make sure that your tarball has the correct files. You can view the contents of a tarball by using:

tar -tvzf filename.tar.gz

Emailing Your Submission

Once you have created the tarball with your submission files, email it to your TA. The email subject line must look like "[EECS 268] SubmissionName":

[EECS 268] Lab 01

Note that the subject should be exactly like the line above. Do not leave out any of the spaces, or the bracket characters ("[" and "]"). In the body of your email, include your name and student ID.

Rubric

Grades will be assigned according to the following criteria:

  • [5pts] Email title, attachment
  • [10pts] Commenting
    • File, author, date, purpose at the top of every file.
    • pre and post conditions for every method signature in the header files
  • [5pts] Formatting
    • Standard indentation
  • [25pts] Pokemon class
    • [5pts] All member variables private
    • [10pts] All set methods defined
    • [10pts] All get methods defined and const
  • [40pts] Colosseum class
    • [20pts] play
      • Game end when either player dies or rounds are over
    • [10pts] userBuild
    • [10pts] attack
  • [10pts] main class
    • Declares stack allocated Pokemon and Colosseum instances to drive the game
    • This does not handle the game, it calls the Colosseum methods to do so
    • Allows the user to repeat as desired
